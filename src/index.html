Back-end com NodeJS

Conceito de Node.js e API REST

🟩 Node.js

Node.js é uma ferramenta que permite rodar JavaScript no servidor.

Com ele, você cria back-end, APIs, sistemas em tempo real, e muito mais.

  1. Permite rodar JS fora do navegador.
  2. Ideal pra criar servidores, APIs e apps rápidos.
  3. Usa o motor V8 do Chrome por baixo dos panos.

🌐 API REST

API REST é um jeito de criar rotas para sistemas conversarem entre si.

Usa métodos HTTP como GET, POST, PUT e DELETE pra manipular dados.

  1. Usa os métodos HTTP (GET, POST, PUT, DELETE...).
  2. Baseada em recursos (ex: /usuarios, /produtos).
  3. Toda comunicação rola via JSON, normalmente.
🌐 Métodos HTTP: Entendendo as Requisições na Web 🌐

Quando navegamos na internet ou interagimos com APIs, usamos os métodos HTTP para comunicar com o back-end. Cada método tem um propósito específico e define a ação que queremos executar no servidor.

Aqui está um resumo dos principais métodos HTTP:

1. GET - Buscar Informações

📌 O que faz?

  • Solicita dados do servidor, mas não altera nada.
  • Como abrir um site ou buscar dados de uma API.

📌 Exemplo:

index.js 
const express = require("express")

const app = express()

app.get("/projects", (request, response) => {
  return response.json(["projeto 1", "projeto 2"])
})

💡 Usado para:

  • Consultar listas de usuários, produtos, artigos etc.
  • Buscar informações sem modificar nada no servidor.
2. POST - Criar um Novo Registro

📌 O que faz?

  • Envia dados para o servidor para criar um novo recurso.
  • Diferente do GET, pode modificar os dados no servidor.

📌 Exemplo:

index.js 
app.post("/projects", (request, response) => {
  return response.json(["projeto 1", "projeto 2", "projeto 3"])
})

💡 Usado para:

  • Criar contas de usuário, posts em um blog, adicionar itens em um banco de dados.
3. PUT - Atualizar um Registro (Completo)

📌 O que faz?

  • Atualiza toda a informação de um recurso existente.
  • Se algum campo estiver ausente, ele pode ser apagado

📌 Exemplo:

index.js 
app.put("/projects/:id", (request, response) => {
  return response.json(["projeto 1", "projeto 2", "projeto 3", "projeto 4"])
})

💡 Usado para:

  • Substituir completamente os dados de um usuário, produto, post etc.
4. PATCH - Atualizar um Registro (Parcial)

📌 O que faz?

  • Atualiza apenas campos específicos de um recurso.
  • Diferente do PUT, os campos não enviados continuam inalterados

📌 Exemplo:

index.js 
app.patch("/projects/:id", (request, response) => {
  return response.json(["projeto 1", "projeto 2", "projeto 3", "projeto 4"])
})

💡 Usado para:

  • Atualizar somente um campo, como mudar a senha ou e-mail de um usuário.
5. DELETE - Apagar um Registro

📌 O que faz?

  • Remove um recurso do servidor permanentemente.

📌 Exemplo:

index.js 
app.delete("/projects/:id", (request, response) => {
  return response.json(["projeto 2", "projeto 3", "projeto 4"])
})

💡 Usado para:

  • Apagar um usuário, remover um produto do catálogo, deletar um post etc.
🚀 Conclusão
Método Ação
GET 🔍 Buscar informações (sem modificar nada)
POST ➕ Criar um novo recurso
PUT 💾 Atualizar um recurso por completo
PATCH 🛠️ Atualizar apenas um campo
DELETE 🗑️ Deletar um recurso
🏷️ Tipos de Parâmetros em Requisições HTTP 🏷️

Quando fazemos requisições para APIs, precisamos passar informações para o servidor. Existem três formas principais de enviar esses dados:

1. Query Params (Parâmetros de Consulta)

📌 O que faz?

  • Passados na URL após o ? e separados por &.
  • Usados para filtrar, ordenar ou paginar resultados

📌 Exemplo:

index.js 
app.get("/projects", (request, response) => {
  const { title, owner } = request.query

  console.log(title)
  console.log(owner)

  return response.json(["projeto 1", "projeto 2"])
})

URL gerada:

arduino 
http://localhost:3333/projects?title=React&owner=Guilherme

💡 Usado para:

  • Filtros em listas (ex: Titulo equivalente a "React").
  • Paginação (?title=React&owner=Guilherme).
2. Route Params (Parâmetros de Rota)

📌 O que faz?

  • Fazem parte da URL, sem ?.
  • Usados para identificar um recurso específico.

📌 Exemplo:

index.js 
app.put("/projects/:id", (request, response) => {
  // const params = request.params
  // console.log(params)
  // { id: '1' }

  const { id } = request.params
  console.log(id)
  // 1

  return response.json(["projeto 1", "projeto 2", "projeto 3", "projeto 4"])
})

URL gerada:

script.js 
http://localhost:3333/projects/1?

💡 Usado para:

  • Buscar uma rota específica (/projects/1).
3. Request Body (Corpo da Requisição)

📌 O que faz?

  • Não vai na URL! É enviado no corpo da requisição.
  • Usado para enviar dados complexos, como JSON.
  • Exige métodos como POST, PUT ou PATCH.

📌 Exemplo:

Por padrão o express não interpreta o que vai ser enviado para ele no formato json. Então para falarmos para o express que a nossa API vai receber dados no formato de json, so precisamos adicionar app.use(express.json()).

⚠️ sempre antes da chamada das rotas e logo apos da definição da variavel app!

script,js 
app.use(express.json())

app.post("/projects", (request, response) => {
  const body = request.body
  
  console.log(body)

  return response.json(["projeto 1", "projeto 2", "projeto 3"])
})

JSON enviado no body:

Body => JSON 
{
  "nome": "Guilherme",
  "email": "guilherme@email.com",
  "idade": 27
}

💡 Usado para:

  • Criar usuários, enviar formulários, atualizar informações
🚀 Conclusão
Tipo de Parâmetro Onde Fica? Exemplo Usado Para
Query Params URL após ? /usuarios?idade=25 Filtros, paginação
Route Params Na URL /usuarios/123 Identificar um recurso
Request Body No corpo da requisição { "nome": "João" } Criar ou atualizar dados
Header Params No cabeçalho da requisição "Authorization": "Bearer token123" Autenticação, metadados
Path Params Na URL, dentro de uma hierarquia /lojas/15/produtos/200 Recursos dentro de categorias
Form Data No corpo da requisição formData.append("foto", file) Upload de arquivos
🧾 Lista esperta dos response.status(n) mais comuns: 🧾

1. (1xx) – Informativos (quase nunca usados no back-end do dia a dia)

🔧 Ex: 100 Continue → usado por navegadores e servidores pra saber se devem continuar a requisição.

👉 Você quase nunca vai usar. Bora pro que interessa.

2. (2xx) – Sucesso (Usado para expressar êxito na requisição)

📌 200 OK:

  • 🔸 Requisição foi processada com sucesso.
  • 🧠 Usa quando: você quer responder com dados ou mensagens normais.
index.js 
return response.status(200).json({ msg: "Usuário encontrado!" })

📌 201 Created:

  • 🔸 Algo foi criado com sucesso (ex: novo usuário, post, comentário...).
  • 🧠 Usa quando: um novo recurso é criado com POST.
index.js 
return response.status(201).json({ msg: "Usuário criado!" })

📌 204 No Content:

  • 🔸 Requisição bem-sucedida, mas sem conteúdo pra retornar.
  • 🧠 Usa quando: deleta algo ou atualiza e não precisa responder com dados.
index.js 
return response.status(204).send()
Código Significado Quando usar
200 OK Quando uma requisição foi processada com sucesso (GET, PUT)
201 Created Quando você cria algo com sucesso (POST, por exemplo)
204 No Content Quando deu tudo certo, mas não tem nada pra retornar
3. (3xx) – Redirecionamentos (mais comuns no front ou servidores como o Nginx)
  • Ex: 301 Moved Permanently, 302 Found
  • 🧠 Quase nunca usados no Express diretamente. Quando precisar redirecionar, pode usar:
index.js 
response.redirect("/nova-pagina")
4. (4xx) – Erro do cliente (o famoso "foi você que errou , não o servidor")

📌 400 Bad Request:

  • 🔸 A requisição veio errada ou malformada.
  • 🧠 Usa quando: dados estão inválidos, campos faltando, etc.
index.js 
return response.status(400).json({ error: "Nome é obrigatório." })

📌 401 Unauthorized:

  • 🔸 O cliente não está autenticado.
  • 🧠 Usa quando: precisa de login/autenticação (JWT, token, etc).
index.js 
return response.status(401).json({ error: "Você precisa estar logado." })

📌 403 Forbidden:

  • 🔸 O cliente está logado, mas não tem permissão.
  • 🧠 Usa quando: usuário tenta acessar algo que ele não pode.
index.js 
return response.status(403).json({ error: "Acesso negado." })

📌 404 Not Found:

  • 🔸 Recurso não encontrado.
  • 🧠 Usa quando: o ID não existe, rota errada, etc.
index.js 
return response.status(403).json({ error: "Acesso negado." })

📌 409 Conflict:

  • 🔸 Conflito de dados (ex: registro já existe).
  • 🧠 Usa quando: email já cadastrado, CPF duplicado, etc.
index.js 
return response.status(404).json({ error: "Usuário não encontrado." })
5. (5xx) – Erros do servidor (a culpa agora é nossa 😅)

📌 500 Internal Server Error:

  • 🔸 Erro interno do servidor. Algo deu ruim.
  • 🧠 Usa quando: estoura uma exceção, erro desconhecido.
index.js 
return response.status(500).json({ error: "Erro inesperado no servidor." })

📌 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout:

  • 🔸 Problemas de infraestrutura (outro serviço caiu, servidor ocupado, etc).
  • 🧠 Mais comuns em apps maiores com múltiplos serviços (tipo microserviços).
6. 🔥 Dica Ninja:

Você pode criar um padrão no seu back-end tipo assim:

index.js 
if (!user) {
  return response.status(404).json({ error: "Usuário não encontrado" })
}

if (!email || !senha) {
  return response.status(400).json({ error: "Campos obrigatórios" })
}

return response.status(200).json({ msg: "Usuário logado com sucesso!" })
🛑 Middleware: O Interceptador de Requisições 🛑

O middleware é como um segurança que fica no meio do caminho entre a requisição do cliente e a resposta do servidor.

Ele pode modificar, bloquear ou encaminhar a requisição antes que ela chegue ao destino final.

📌 Definição:

Um middleware é uma função que intercepta a requisição antes dela chegar à rota final. Podendo assim:

  • Modificar a requisição ou a resposta.
  • Bloquear o acesso caso não cumpra algum critério.
  • Registrar logs e métricas.
  • Adicionar informações extras na requisição

Ele e utilizada principalmente quando houver a nessecidade de executar um mesmo trexo de código em multiplas rotas.

×

Uma middleware e composta por:

  • function functionName(request, response, next):
    • request: Este é um objeto que contém informações sobre a requisição HTTP que o servidor recebeu. Isso inclui coisas como o método HTTP (GET, POST, PUT, DELETE, etc.), a URL solicitada, os headers da requisição, o corpo da requisição (se houver), e muito mais.
    • response: Este é um objeto que você usa para enviar uma resposta de volta ao cliente que fez a requisição. Ele permite definir o status code da resposta, os headers da resposta, e o corpo da resposta.
    • next: Esta é uma função muito importante. Quando um middleware termina de executar sua lógica, ele precisa chamar a função next() para passar o controle para o próximo middleware na cadeia ou para a rota final que irá lidar com a requisição. Se next() não for chamado, a requisição ficará "presa" no middleware e a resposta nunca será enviada.
Fluxo da Requisição com Middleware 
[ Cliente ] → [ Middleware 1 ] → [ Middleware 2 ] → [ Rota Final ]
Exemplos de como criar Middlewares

📌 Criando um middleware que registrar informações sobre cada requisições recebidas:

index.js 
function logRequests(request, response, next) {
  const { method, url } = request

  const logLabel = `[${method.toUpperCase()}] ${url}`

  console.log(logLabel)

  console.time("logLabel")

  next()

  console.timeEnd("logLabel")
}

💡 Em resumo, o que esse middleware faz é:

  • Registrar informações sobre cada requisição recebida: Ele loga o método HTTP e a URL da requisição no console.
  • Medir o tempo de processamento da requisição: Ele inicia um cronômetro antes de passar o controle para o próximo middleware/rota e para o cronômetro depois que a rota e quaisquer middlewares subsequentes terminarem de executar. Isso ajuda a identificar possíveis gargalos de desempenho na sua aplicação.
  • Passar o controle adiante: Ao chamar next(), ele garante que a requisição continue seu fluxo normal através da sua aplicação.

📌 Criando um middleware que executa a validação do ID:

index.js 
function validateProjectID(request, response, next) {
  const { id } = request.params
  console.log("1")

  if (!isUuid(id)) {
    return response.status(400).json({ error: "Invalid project ID." })
  }

  return next()
}

💡 Em resumo, o que esse middleware faz é:

  • Se o id não for um UUID válido, uma resposta de erro 400 é enviada e a função retorna, impedindo a chamada de next().
  • Se o id for um UUID válido, o middleware simplesmente chama next(), permitindo que a requisição continue para a próxima função na cadeia (que provavelmente será a rota que realmente busca ou manipula o projeto com o ID fornecido).

📜 Metodos de uso para um Middleware!

Os middlewares podem ter diferentes funções, dependendo do que queremos fazer:

1. Middleware Global
  • Aplica-se a todas as requisições.
  • Exemplo: Registro de logs, CORS, parsing de JSON.
index.js 
// Sem rota especifica
function logRequests(request, response, next) {...} 

app.use(logRequests)

// Rota sendo especificada

app.use("/projects/:id", logRequests)

💡 O que acontece aqui?

  1. Nese metodo nos temos o middleware sendo usado da seguinte forma utiliza o app.use(logRequests) para executar sempre que houver alguma requisição, sendo aplicado em qualquer rota.
  2. Nese metodo nos temos o middleware sendo aplicado em uma rota especifica,app.use("/projects/:id", logRequests).
  3. Em ambos os metodos pode ser aplicado mais de uma Middleware no momento da chamada, as separando por uma virgula(,) no momento da chamada.
2. Middleware de Rota Específica
  • Funciona apenas em determinadas rotas.
  • Exemplo: Autenticação antes de acessar uma área restrita.

📌 Exemplo:

index.js 
function logRequests(request, response, next) {...}
function validateProjectID(request, response, next) {...}

app.use("/projects/:id", logRequests, validateProjectID)

💡 O que acontece aqui?

Nese metodo nos temos o middleware sendo aplicado dentro de uma rota específica, podendo ser aplicado mais de uma middleware para a rota chamada app.use("/projects/:id", logRequests, validateProjectID).

3. Middleware de Tratamento de Erros
  • Captura e lida com erros nas requisições.
  • Deve ter quatro parâmetros (err, req, res, next).

📌 Exemplo:

index.js 
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).send("Algo deu errado!");
});
🚀 Conclusão
Tipo Descrição Exemplo de Uso
Global Aplica-se a todas as requisições app.use(logRequests) e app.use(logRequests, validateProjectID)
Rota Específica Funciona apenas em algumas rotas app.delete("/projects/:id", checkAuth, ...) e app.use("/projects/:id", validateProjectID, ...)
Erro Trata erros na aplicação app.use((err, req, res, next) => { ... })
